Namaa — CLIENT-SIDE REACT APPLICATION DOCUMENTATION
=============================================================

Welcome to the official technical documentation for the client-side (user-facing) React application of the Namaa graduation project. This document serves as a complete reference for frontend developers, detailing the application's architecture, page-level user flows, global state providers, authentication workflows, and UI interactions to ensure a source-code-independent understanding of the system.

________________________________________________________________________________


1. Project Overview
-------------------

Namaa is a premium, AI-driven business analytics platform designed specifically for Small and Medium Businesses (SMBs). Its core mission is to empower business owners to extract actionable intelligence from their relational databases (such as SQL Server and PostgreSQL) using natural language querying and AI algorithms.


1.1 Key Capabilities:
• Interactive Pricing & Estimator: Browse package tiers, select individual AI capabilities, fine-tune commitments, and perform real-time pricing calculations.
• Onboarding & Database Setup: Secure forms that guide users through linking their external databases with Namaa processing nodes.
• Client Workspace Dashboard: Displays token consumption metrics, active subscription terms, auto-renewal controls, and data synchronization health.
• Billing & Account Management: Manages secure credit card storage, invoice downloads, security sessions, and personal profiles.

4. Component Architecture
-------------------------
This section documents the non-page, structural UI components used as global layouts, navigation panels, route guards, or reusable cards.

4.1 Route Guards

4.1.1 Protected (Protected.jsx)
The Protected (Protected.jsx) component belongs to the Route Guard category. Its primary purpose is to restricts access to client workspace routes for unauthenticated users. It accepts properties including: children (React Node): The secure component to mount if authorized., and maintains no internal state. In terms of behavior, the component reads userToken and loading states from userContext. If loading, renders a loader spinner; if authenticated, mounts children; if unauthorized, redirects the client to /login. It is integrated and used in Secure routes wrappers in App.jsx and has dependencies on react-router-dom, userContext.

4.2 Layout Components

4.2.1 Layout (Layout.jsx)
The Layout (Layout.jsx) component belongs to the Layout Container category. Its primary purpose is to serve as the primary public root container. Renders global branding headers, footers, and wraps around public marketing pages. It does not accept any external props, and maintains no internal state. In terms of behavior, the component displays <NavBar /> at the top, inserts dynamic nested route pages inside a centered React Router <Outlet />, and mounts <Footer /> at the bottom. It is integrated and used in Root routes in App.jsx and has dependencies on react-router-dom, NavBar, Footer.

4.2.2 DashboardLayout (DashboardLayout.jsx)
The DashboardLayout (DashboardLayout.jsx) component belongs to the Layout Container category. Its primary purpose is to serve as the main client workspace container shell. Renders sidebar panels and toggles navigation menus on small screens. It does not accept any external props, and maintains internal state variables including: isSidebarOpen (Boolean): Controls sidebar expansion states on mobile screen layouts. In terms of behavior, the component displays navigation selectors on the left side, active profile badges in the header, and details workspace modules inside <Outlet /> on the right side. It is integrated and used in Dashboard protected sub-routes in App.jsx and has dependencies on react-router-dom, lucide-react, userContext.

4.2.3 ProfileLayout (ProfileLayout.jsx)
The ProfileLayout (ProfileLayout.jsx) component belongs to the Layout Container category. Its primary purpose is to parent sidebar wrapper for account settings sub-routes, managing avatar uploads and sign-out processes. It does not accept any external props, and maintains internal state variables including: userData (Object): Local cache of active settings profile; loading (Boolean): Form spinner indicator. In terms of behavior, the component fetches user profile on mount; Clicking the avatar selects a image file, converts it via FileReader, and uploads it as multipart/form-data; * Renders setting sub-pages inside <Outlet />. It is integrated and used in Settings nested routes in App.jsx and has dependencies on react-router-dom, userContext, api, toast.

4.2.4 NavBar (NavBar.jsx)
The NavBar (NavBar.jsx) component belongs to the Navigation Panel category. Its primary purpose is to top header navigation showing branding logs, public page selectors, active cart count, and user control dropdowns. It does not accept any external props, and maintains internal state variables including: isMenuOpen (Boolean): Collapses nav elements on mobile screens. In terms of behavior, the component monitors the global authentication state from userContext to display user profiles or show login actions, and reads items count from CartContext. It is integrated and used in Layout and has dependencies on react-router-dom, userContext, CartContext.

4.2.5 Footer (Footer.jsx)
The Footer (Footer.jsx) component belongs to the Static Layout category. Its primary purpose is to standard bottom footer providing links, branding assets, copyright strings, and social icons. It does not accept any external props, and maintains no internal state. In terms of behavior, the component renders static styling links grids. It is integrated and used in Layout and has dependencies on react-router-dom.

4.3 Shared / Reusable UI Components

4.3.1 ProfileSubscription (ProfileSubscription.jsx)
The ProfileSubscription (ProfileSubscription.jsx) component belongs to the Reusable UI Card category. Its primary purpose is to renders active subscription status summaries, charge notifications, and features tables inside settings views. It does not accept any external props, and maintains no internal state. In terms of behavior, the component renders a details dashboard, and navigates clients to /pricing upon clicking "Upgrade Plan". It is integrated and used in Profile settings panels and has dependencies on react-router-dom.

4.3.2 Complete (Complete.jsx)
The Complete (Complete.jsx) component belongs to the Reusable UI Card category. Its primary purpose is to shows onboarding checkout status indicators and guides new users to databases credentials setups. It does not accept any external props, and maintains internal state variables including: completeData (Array): Lists recently purchased services. In terms of behavior, the component fetches transaction lists on mount, lists completed payment checkmarks, and offers a button to navigate to /Setup-page. It is integrated and used in Onboarding views (/complete-data) and has dependencies on react-router-dom, userContext, api, toast.

6.1 Flow A: Guest Discovery & Pricing (Visitor Journey)

6.1.1 Homepage (Home.jsx)
This page component is mapped to the route path /home or /. Its primary purpose is to serve as the primary public entry point introducing Namaa, categories, and testimonials. It does not require any external properties or maintain internal state. The user interface layout comprises sections such as: Hero banner with glowing gradients; Categories navigation bar with service cards; * Client reviews marquee area. Its core technical behavior includes: Fetch reviews on mount. If reviews.length > 3, duplicates the array to animate a continuous CSS horizontal marquee scrolling loop; Tab click triggers class changes for smooth element fade-ins. The component handles data display and API integration by: Hits GET /Reviews/landing-page on mount; Renders names, ratings, text, and avatars. The standard user flow proceeds as follows: 1. User lands on page and views testimonials; 2. Clicks service category tabs to check details; 3. Clicks "Explore Features" link to go to Features Catalog. Finally, it depends on: react-router-dom, api.

6.1.2 Features Catalog (Features.jsx)
This page component is mapped to the route path /features. Its primary purpose is to lists all available AI services. It does not require any external properties or maintain internal state. The user interface layout comprises sections such as: Grid directory of service cards with title, description, and price badge; Top filter bar with search inputs. Its core technical behavior includes: Fetches services on mount; Uses clientside filter rules to filter services array by searchTerm. The component handles data display and API integration by: Calls GET /Services on mount; Renders service items. It handles user interactions and states including: Displays a loading state until search resolves. The standard user flow proceeds as follows: 1. User browses card lists; 2. Types search filter keyword; 3. Clicks card to go to Feature Details. Finally, it depends on: react-router-dom, api.

6.1.3 Feature Details (FeatureDetails.jsx)
This page component is mapped to the route path /feature-details/:id. Its primary purpose is to detailed specification sheet for a selected service. It does not require any external properties or maintain internal state. The user interface layout comprises sections such as: Left Column: Spec sheets (token limits, response times); Right Column: Checklists, Use Cases, reviews. Its core technical behavior includes: Extract id from path parameters; Fetches service spec details on mount. The component handles data display and API integration by: Hits GET /Services/${id} on mount. The standard user flow proceeds as follows: 1. User reads benefits checklists; 2. Clicks "See Pricing" to go to Pricing packages constructor. Finally, it depends on: react-router-dom, api.

6.1.4 Pricing & Bundles Constructor (Pricing.jsx)
This page component is mapped to the route path /pricing. Its primary purpose is to plan comparer and custom services calculator. It does not require any external properties or maintain internal state. The user interface layout comprises sections such as: Top: 3D packages carousel cards; Bottom: Customizable services checklist. Its core technical behavior includes: Left/Right arrows shift carousel active cards, applying CSS scale-up transitions; Cart add action saves details to context cart values. The component handles data display and API integration by: Hits GET /Packages (packages) and GET /Services/cards (estimator list). It handles user interactions and states including: Updates item counts badge in navbar dynamically. The standard user flow proceeds as follows: 1. User compares package options in 3D carousel; 2. Toggles services checkbox add-ons; 3. Clicks cart icon button to review selections. Finally, it depends on: react-router-dom, userContext, CartContext, api.

6.1.5 Shopping Cart (Cart.jsx)
This page component is mapped to the route path /cart. Its primary purpose is to pro-rated cost calculator and checkout interface. It does not require any external properties or maintain internal state. The user interface layout comprises sections such as: Left: Cart items checklist with duration and token count dropdown selectors; Right: Pro-rated receipts summary box. Its core technical behavior includes: Cart calculations run on client: Monthly = Base Duration + Selected Token Cost; Guest items in local storage merge to database cart on login. The component handles data display and API integration by: Hitting GET /Cart to list items; Hitting GET /Orders/discount-codes/validate?code=${promoCode} to validate codes; * Hitting POST /Orders/services to checkout. It handles user interactions and states including: Proceeds to payment. Locks page with loading spinners, redirecting user to Stripe payment portal. The standard user flow proceeds as follows: 1. Adjusts commitment duration, token limits; 2. Applies discount coupons; 3. Clicking checkout redirects to Stripe. Finally, it depends on: react-router-dom, userContext, CartContext, api, toast.

7. Authentication & Security Flows
----------------------------------

Authentication is centered around JSON Web Tokens (JWT) stored in the browser and updated silently during sessions.


7.1 Google OAuth Callback Pipeline
Integrated via the GoogleCallback.jsx component to handle third-party login results:
7.1.1 State Decoding: On mount, it parses code and state parameters from window.location.search.
7.1.2 Context Resolution: The state parameter contains a colon-separated string: <flow>:<redirectPath> (e.g. login:/cart or register:/home).
7.1.3 API Handshake:
• If the flow is registration, it calls POST /Auth/google/register.
• If the flow is login, it calls POST /Auth/google/login.
• The payload contains the authorization code and a matching redirectUri.
7.1.4 Session Activation: On success, it writes the returned JWT to localStorage, updates the userContext state, and redirects the user to the encoded redirectPath.


7.2 Axios Interceptors and Silent Token Refresh
The Axios configuration in src/api.js manages authorization headers and automatic token updates:
7.2.1 Request Interceptor: Automatically injects Authorization: Bearer <token> into outgoing headers if a token exists in localStorage.
7.2.2 Response Interceptor & Silent Refresh:
• Detects 401 Unauthorized responses.
• If a refresh process is not already running, it flags isRefreshing = true and calls POST /Auth/refresh, passing the stored refresh token.
• While the refresh request is active, other failing requests are added to a failedQueue array.
• Success Handler: Resolves the failedQueue with the new token, replays the requests, and updates localStorage.
• Failure Handler: Clears the queue, removes tokens from localStorage, resets contexts, and redirects the user to /login.

12. Deployment (Vercel)
-----------------------

The client application is optimized for deployment to Vercel as a static Single Page Application (SPA).


12.1 Wildcard Rewrites (vercel.json)
To support client-side history routing (React Router) and prevent 404 Not Found errors when users directly reload dashboard paths, a vercel.json file must be present in the project root directory containing:

{
 "rewrites": [
 { "source": "/(.*)", "destination": "/index.html" }
 ]
}

• Purpose: Reroutes all direct navigation requests back to the entry point (index.html) so React Router can process the path client-side.


12.2 Step-by-Step Deployment Process
1. Push your local workspace changes to a Git repository (GitHub, GitLab, or Bitbucket).
2. Log in to your Vercel Dashboard and click Add New > Project.
3. Import the repository matching the workspace.
4. Configure the Build & Development Settings:
• Framework Preset: Vite
• Build Command: npm run build
• Output Directory: dist
5. Configure the Environment Variables:
• Add VITE_API_BASE_URL with your production API backend endpoint (e.g., https://api.deeb-ai.com).
6. Click Deploy. Vercel will build the bundles, optimize files, and generate live hosting links.

14. Performance Optimization, Known Issues & Future Improvements
----------------------------------------------------------------


14.1 Performance Optimization
• Lazy Image Loading: Content illustration assets use native loading="lazy" attributes to prioritize core component rendering.
• Debounced API Updates: Toggle controls for notification preferences use a 500ms debounce window to prevent rapid, successive requests.
• Optimistic UI Updates: Toggles for subscription auto-renewals update immediately in the UI and revert only if the backend API requests fail.


14.2 Known Issues
• Duplicate Guest Cart Entries: If a guest user adds the same service multiple times to their local cart, logging in will send duplicate requests, resulting in duplicate database entries.
• Avatar Image Caching: Browser caching can sometimes prevent updated avatar images from rendering immediately. To resolve this, the profile picture path appends a timestamp query parameter (?t=timestamp) to force cache updates.
• Payment Redirect Delays: The redirect to Stripe's payment gateway can take 2-3 seconds during peak server loads. A loading spinner locks the UI during this process to prevent multiple checkout submissions.


14.3 Future Improvements
• Local State Caching: Integrate React Query (TanStack Query) to cache API responses (such as services catalogs or active sessions) and reduce redundant network calls.
• Bulk Cart Syncing: Replace individual requests for guest cart syncing with a single bulk import endpoint to improve login performance.
• Real-time Session Updates: Use WebSockets (SignalR) to update active session lists and device trust statuses in real time.
